## シナリオ

あなたは **FitLoop 株式会社** のプロダクト改善委員会の一員として、来年度（2026年度）の **運動継続支援プログラム** を経営層に提案することになった。

FitLoop はスマートフォン向けフィットネスアプリを開発・運営するスタートアップで、ユーザーがワークアウト記録・トレーナー相談・バッジ獲得などの機能を使える。

経営層からの依頼は以下のとおり。

> 「今年度より**継続率**を改善したい。
>  ただし、追加で投入できる予算と人員は限られている。
>  **どのユーザー層**に、**どんな支援**を、**どのタイミング**で行うべきかを、
>  過去3四半期分のユーザーデータから根拠を示して提案してほしい。」

プロダクトオーナーからの追加コメント：

> 「『継続率を上げる』だけだと、ザルなKPIになる。
>  元から意欲の高いユーザーは放っておいても続ける。本当に効くのは『あと一押しで変わる層』への介入だ。
>  そこをデータで切り出してほしい。」

このシナリオに対し、Phase2 を通じて分析計画を組み立てていく。

## 共通データ仕様

各コマで同じ `df` を使い続ける。コマ8の冒頭で1度生成し、以降は再利用する。

```python
import pandas as pd
import numpy as np
import matplotlib
matplotlib.rcParams['font.family'] = 'IPAGothic'
import matplotlib.pyplot as plt

RANDOM_STATE = 42
rng = np.random.default_rng(RANDOM_STATE)

n = 600  # 過去3四半期 × 約200人/四半期
df = pd.DataFrame({
    'user_id':           np.arange(1, n + 1),
    'register_quarter':  rng.choice(['2023Q3', '2023Q4', '2024Q1'], n, p=[0.32, 0.34, 0.34]),
    'plan_type':         rng.choice(['ベーシック', 'スタンダード', 'プレミアム', 'トライアル'], n),
    'workout_freq_m1':   np.round(rng.normal(3.2, 1.2, n).clip(0, 7), 1),   # 週あたり回数（1ヶ月目）
    'workout_freq_m3':   np.round(rng.normal(2.8, 1.4, n).clip(0, 7), 1),   # 週あたり回数（3ヶ月目）
    'login_rate_m1':     np.round(rng.beta(8, 2, n), 2),                     # ログイン率（1ヶ月目）
    'login_rate_m3':     np.round(rng.beta(7, 3, n), 2),                     # ログイン率（3ヶ月目）
    'achievement_count': rng.poisson(1.5, n),                                 # バッジ獲得数
    'trainer_session':   rng.binomial(1, 0.45, n),                            # トレーナー相談利用フラグ
    'push_opt_in':       rng.binomial(1, 0.35, n),                            # プッシュ通知ONフラグ
    'continued':         rng.binomial(1, 0.70, n),                            # 6ヶ月後も継続中フラグ
})

# 「効くはずの変数」を弱く埋め込む（分析する価値が出るように）
boost = (
    0.12 * (df['login_rate_m1'] > 0.85).astype(int)
    + 0.10 * (df['push_opt_in'] == 1).astype(int)
    + 0.08 * (df['achievement_count'] >= 2).astype(int)
)
df['continued'] = ((df['continued'] + rng.binomial(1, boost.clip(0, 1))) >= 1).astype(int)

plan_price_map = {'ベーシック': 980, 'スタンダード': 1980, 'プレミアム': 2980, 'トライアル': 0}
df['plan_price'] = df['plan_type'].map(plan_price_map)
```

> **注意**：このデータは仮想生成。学生に「実データのつもりで分析するが、結論を実サービスに持ち込まない」ことを明示する。

## コマ6 担当パーツ：問いの初期設計

### お題

経営層・プロダクトオーナーのコメントを、**5個の小さな問い**に分解せよ。

### やること

1. 経営層のメッセージから「KPI候補」を**3つ以上**挙げる
   （継続率？／平均継続日数？／早期離脱の防止？／アクティブ率？など）
2. ロジックツリーで大きな問いを 5 個の小問いに分解する（Mermaid または テキスト可）
3. 各小問いに対して、以下を1行で書く：
   - 答えが出たら **何が変わるか**（So What?）
   - **答えやすさ**（◎ ／ ○ ／ △ ／ ×）
   - **意思決定への効きやすさ**（◎ ／ ○ ／ △ ／ ×）
4. 5個の中から、Phase3 以降で深掘りする **1個** を仮選定する

### 検討させる視点

- 「継続率を上げる」を**そのまま**KPIにすると何が困るか？
- 問いを **個人レベル / セグメントレベル / プログラム全体** のどこに置くかで、必要なデータと手法が変わる
- プロダクトオーナーの「あと一押しで変わる層」は、どの問いに対応するか？

### 成果物パーツ

## コマ7 担当パーツ：分析ストーリーの設計

### お題

仮選定した問いに対する **「経営会議で見せる最終1スライド」** をテキストで設計せよ。

### やること

1. 結論文を1文で書く（仮でよい、分析後に書き換えてOK）

2. その結論を支える根拠を **3つ** 用意する（どんな数字・グラフが出れば結論が言えるか）

### 検討させる視点

- 「数字 → 意味 → 施策」の3層が揃っているか？
- 結論が「だから何？（So What?）」に耐えるか？
- 経営層が **最初に知りたいのはどの順番か**（時系列 vs 結論ファースト）？

## コマ8〜9 担当パーツ：ユーザーセグメンテーション

### お題

「あと一押しで変わる層」を**データで定義**せよ。

### やること

1. 以下の手法候補から **2つ以上** を実際に試し、結果を比較する

   | 手法候補 | このプロジェクトでの解釈 | 学べるコマ |
   |---------|----------------------|----------|
   | RFM 分析 | R=ログイン率, F=ワークアウト頻度, M=バッジ数 として3軸スコア化 | コマ8 |
   | コホート分析 | 登録四半期 × 月 で継続率の経時変化 | コマ8 |
   | ピボット集計 | プラン × ログイン率帯 × 継続率 など多変量集計 | コマ9 |
   | （発展）k-means | 多変量グルーピング（Phase3 で再検討可） | Phase3 |

2. 「あと一押し層」の **定義文** を1〜2行で書く
   （例：`login_rate_m1 ∈ [0.5, 0.75]` かつ `achievement_count == 1` かつ `push_opt_in == 0` のユーザー）
3. その層が全体の **何%** いるか、**継続率は全体平均と比べてどうか** を数値で示す
4. このセグメントへの **初期施策案** を1つ書く
5. **財務インパクトを試算する**
   - あと一押し層の平均月額料金（`plan_price` の平均）を求める
   - あと一押し層が**全員離脱した場合の月次損失**を試算する（人数 × 平均月額）
   - 選んだ施策を月次予算40万円の範囲で**何人に届けられるか**を計算する
   - 継続率改善による**期待追加売上**と施策コストを比較し、ROIを求める
     ```
     期待追加売上 = 継続率改善幅(pt) × あと一押し層の人数 × 平均月額 × 6ヶ月
     ROI = (期待追加売上 − 施策コスト) ÷ 施策コスト × 100
     ```

### 検討させる視点（手法選定の根拠）

| 検討ポイント | 学生が答えるべきこと |
|------------|-------------------|
| なぜその手法を選んだか | 問い（=あと一押し層の特定）に対する適合性 |
| 他の手法を選ばなかった理由 | データ・粒度・解釈しやすさのどれが効いたか |
| セグメント数の決め方 | 経営層に説明できる粒度か？（多すぎ／少なすぎ） |
| RFM のスコア区切り | 分位数 vs 固定閾値、どちらが今回の問いに合うか |

### 成果物パーツ

- `Q03_segmentation.ipynb`：手法2つの実装 + セグメント定義 + 数値根拠 + 初期施策案

## コマ10〜11 担当パーツ：時系列傾向と施策の効果検証

### お題

セグメンテーションで定義した「あと一押し層」に対する施策の **効果検証計画** を設計せよ。

### やること

1. **時系列の観点**：以下から1つ以上を実装する
   - 登録四半期 × 継続率の推移（四半期効果はあるか）
   - 1ヶ月目ログイン率と3ヶ月目ワークアウト頻度の関係（早期シグナルとして使えるか）
   - プッシュ通知ON × 継続率 の四半期別比較

2. **施策の効果検証**：A/B テスト設計書を書く
   - A群・B群の定義（誰を、どう割り当てるか）
   - 評価指標と判定基準（何%差で「効果あり」とするか）
   - 必要サンプルサイズの目安
   - **選択バイアス**への対処方針（既存の `push_opt_in` フラグはランダム割り当てではない点に注意）
   - **費用対効果の試算**（施策コスト・期待追加売上・ROI・予算40万円内に収まるか）

3. 既存データで二群比率の検定を実施し、観察された差と p 値を報告する
   （観察データなので「相関であって因果ではない」点を必ず明記）

### 検討させる視点

- **時系列**で見るべき粒度は？（日次・週次・月次・四半期）データ点数とのバランス
- **A/B テスト**で「やってはいけない設計」は何か？（事後割り当て・自己選抜・対照群なし）
- 「既にプッシュ通知をONにしている人」の効果と、「これからONにさせる人」の予測効果は同じか？

### 成果物パーツ

- `Q04_timeseries_and_ab.ipynb`：時系列分析の結果 + A/B 設計書 + 既存データでの検定結果
